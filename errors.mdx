PermitCore returns an [RFC 9457](https://www.rfc-editor.org/rfc/rfc9457)
`application/problem+json` body for every non-2xx response:

```json
{
  "type": "https://docs.permitcore.io/errors/missing-credentials",
  "title": "Missing Credentials",
  "status": 401,
  "detail": "Missing Authorization: Bearer <key> header.",
  "instance": "/v1/permits",
  "request_id": "req_1f07fb45782b445e9f6312ba23921234"
}
```

**Branch your error handling on `type`** (a stable URI), not on `title` or
`detail` (human-readable, may change). `status` mirrors the HTTP status;
`instance` is the request path; `request_id` also appears as the `x-request-id`
response header — include it in support requests.

## Error catalog

The `type` URI ends in the slug shown below; each links to a page explaining the
error and how to fix it.

### 401 Unauthorized

| `type` slug | Cause | Retry? |
|-------------|-------|--------|
| [`missing-credentials`](/errors/missing-credentials) | `Authorization` header absent | No — fix client and resubmit |
| [`invalid-credentials`](/errors/invalid-credentials) | Key is malformed, revoked, or unknown | No — rotate key via dashboard |

Auth is checked before the resource, so a bad key returns `invalid-credentials`
even on an unknown path or jurisdiction.

### 400 / 422 Bad request

| `type` slug | Cause | Retry? |
|-------------|-------|--------|
| [`invalid-request`](/errors/invalid-request) | A query parameter failed validation (response carries an `errors` array) | No — fix the parameter |
| [`invalid-bbox`](/errors/invalid-bbox) | `bbox` is not `minLon,minLat,maxLon,maxLat` | No — fix the value |
| [`missing-jurisdiction`](/errors/missing-jurisdiction) | A required geographic filter was omitted | No — add `jurisdiction=` |
| [`invalid-canonical-uid`](/errors/invalid-canonical-uid) | `canonical_uid` path segment is malformed / not URL-encoded | No — encode + fix shape |
| [`invalid-target-segment`](/errors/invalid-target-segment) | Unknown `target_segment` value (whitelist in `detail`) | No — use a canonical segment |
| [`unknown-cohort`](/errors/unknown-cohort) | Unknown per-manufacturer `cohort` basename on `/v1/permits/target` | No — use a known cohort |
| [`filter-required`](/errors/filter-required) | Rollup submitted with no narrowing filter | No — add a filter |
| [`group-cardinality-exceeded`](/errors/group-cardinality-exceeded) | Grouping would return too many rows | No — narrow / coarsen `group_by` |

### 403 Forbidden

| `type` slug | Cause | Retry? |
|-------------|-------|--------|
| [`missing-scope`](/errors/missing-scope) | Key is valid but lacks the scope required for this endpoint | No — request a key with the needed scope |

### 404 Not Found

| `type` slug | Cause | Retry? |
|-------------|-------|--------|
| [`not-found`](/errors/not-found) | Path or resource doesn't exist, OR the jurisdiction is DEPLOY-PENDING | No — verify slug + endpoint |
| [`unknown-jurisdiction`](/errors/unknown-jurisdiction) | Jurisdiction slug not in the `jurisdictions` table | No — use a canonical slug |
| [`permit-not-found`](/errors/permit-not-found) | No permit matched the `permit_id` / `canonical_uid` | No — verify the identifier |
| [`api-key-not-found`](/errors/api-key-not-found) | Referenced key prefix not owned by this account | No — list your keys |
| [`export-not-found`](/errors/export-not-found) | Unknown `export_id` | No — verify the id |

### 429 Too Many Requests

| `type` slug | Cause | Retry? |
|-------------|-------|--------|
| [`monthly-requests-quota-exceeded`](/errors/monthly-requests-quota-exceeded) / [`daily-requests-quota-exceeded`](/errors/daily-requests-quota-exceeded) | Request quota exhausted (see [Rate limits](/concepts/rate-limits)) | Yes — respect `Retry-After` |
| [`monthly-exports-quota-exceeded`](/errors/monthly-exports-quota-exceeded) / [`daily-exports-quota-exceeded`](/errors/daily-exports-quota-exceeded) | Export-row quota exhausted | Yes — respect `Retry-After` |
| [`export-concurrency-exceeded`](/errors/export-concurrency-exceeded) | Too many concurrent export jobs in flight | Yes — wait for one to finish |

The emitted quota slug follows the pattern
`{daily\|monthly}-{requests\|exports}-quota-exceeded`. When `Retry-After` is large
(multiple hours), the quota has been exhausted for the window. Bring the key up a
tier or wait for it to reset. **Reset is a trailing 30-day rolling window, not a
calendar month** — monthly quotas restore *progressively* as usage ages out (the
`reset_at` field / `X-RateLimit-Reset` header marks the next freed slot). Daily
quotas restore in full at the next UTC midnight. See [Rate limits](/concepts/rate-limits).

### 5xx Server errors

Retry 5xx on the HTTP `status` (below) with exponential backoff. The `type` slug
is stable for these too.

| HTTP | `type` slug | Cause | Retry? |
|------|-------------|-------|--------|
| `500` | [`internal-error`](/errors/internal-error) | Unexpected pipeline issue | Yes — exponential backoff, alert oncall after 3 retries |
| `503` | [`service-unavailable`](/errors/service-unavailable) | Origin marked unhealthy | Yes — exponential backoff |
| `504` | [`query-timeout`](/errors/query-timeout) | Query exceeded the server-side statement timeout | Yes — narrow filters, then retry |

## Retry guidance

Retryable status codes: `429` (respect `Retry-After`), `500`, `502`, `503`,
`504`. All 4xx codes other than `429` are application errors and should NOT
be retried — fix the request and resubmit.

```js Node.js
const RETRYABLE_STATUS = new Set([429, 500, 502, 503, 504]);

class PermitcoreError extends Error {
  constructor(status, type, detail) {
    super(detail);
    this.status = status;
    // Stable identifier to branch on — e.g. ".../errors/invalid-credentials".
    this.type = type;
  }
}

async function callPermitcore(path) {
  const res = await fetch(`https://api.permitcore.io${path}`, {
    headers: {
      Authorization: `Bearer ${process.env.PERMITCORE_API_KEY}`,
      Accept: "application/json",
    },
  });
  if (!res.ok) {
    // RFC 9457 problem+json: { type, title, status, detail, instance, request_id }
    const body = await res.json().catch(() => ({}));
    throw new PermitcoreError(res.status, body?.type, body?.detail);
  }
  return res.json();
}

async function withRetries(fn, maxAttempts = 4) {
  let lastErr;
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return await fn();
    } catch (err) {
      lastErr = err;
      const retryable = err instanceof PermitcoreError && RETRYABLE_STATUS.has(err.status);
      if (!retryable || attempt === maxAttempts) throw err;
      const delay = Math.min(2 ** attempt * 1000, 30000);
      await new Promise((r) => setTimeout(r, delay));
    }
  }
  throw lastErr;
}

// Usage:
const data = await withRetries(() => callPermitcore("/v1/jurisdictions/nyc/cohorts/distribution"));
```

## Diagnosing in production

- Every response includes a `x-request-id` header (UUID).
  Include it in any support ticket — we can trace per-request.
- Vercel deploy logs surface `[cohort-distribution] <slug>: <error_type>`
  for any caching layer failures.
- For programmatic monitoring, the future `/v1/keys/{prefix}/usage`
  endpoint includes a `recent_requests` array with per-request status.
